# Rules for ÀṢẸ Development

- Always use the `gpt-image-2` model for all image, sprite, and background art generations.
- Never use Gemini or any other image models for generating assets.
- **Syntax Check**: Before committing or deploying, always verify modified JavaScript files using `node --check <path_to_file>` to catch syntax errors like unclosed blocks or brackets.
- **Events ("?" nodes)**: Story events live in `src/data/events.js` and must be original to the ÀṢẸ world (Sunken Market / Brass Archive / Static Crown) — never reskins of Slay the Spire events. Each event has an `art` id; the view (`eventArt()` in `game.js`) shows `assets/event-art/<art>.png`, falling back to a committed `.svg` placeholder, then the generic "?" glyph. Generate art with `node tools/gen-event-art.js` (reads `tools/events.manifest.json`); `--dry-run` writes SVG placeholders with ZERO deps.
- **First-play tutorial**: `src/ui/tutorial.js` (`CombatTutorial`) is a single top-pinned coaching banner shown once, on the first-ever combat, gated by `meta.tutorialDone` (set true when finished OR skipped). Started from `beginCombat` in `game.js` ~1700ms after mount (timed to clear the deferred Battle-Start-then-opening-draw open). It reads live combat state by chaining `combat.onUpdate` (never mutates game logic) and never blocks the board. Six steps: hand+Àṣẹ intro, enemy intent, play a Block skill, play an Attack card, end turn, wrap-up — each of steps 1-5 adds a `.tut-highlight` class (a bright pulsing shine — `filter: brightness()/saturate()` + glow `box-shadow` via `@keyframes tutShine`, deliberately not a static outline/box) to the relevant live DOM node by selector, reapplied every `onUpdate()` since combatView rebuilds those nodes each render. The enemy-intent step also sets `align: 'left'` on its step config (toggles `.tut-banner.tut-left`) so the banner doesn't overlap the intent pill it's pointing at on narrow/landscape-phone widths — verified no bounding-box overlap at 812×375, 700×780, 390×780. The block/attack steps find the first matching card in hand by `uid` and advance only when that uid leaves the hand (falls back to "any card played" if no match exists, ~2-3% chance on a 5-card starter draw, so it can't soft-lock). `game.js`'s `startMonster()` forces the run's first-ever monster fight to `['husk_drone']` when `!meta.tutorialDone` so the guaranteed turn-1 attack intent makes the "foe is about to strike" step always true. Re-test by clearing `localStorage`. Styles under the "first-play tutorial" block in `styles.css`.
- **Auto-Update Learnings**: On every action/task, if you discover a project-specific gotcha, solve a debugging issue, or establish a new convention/pattern, you must immediately update `CLAUDE.md` and `.agents/AGENTS.md` to persist this learning.
- **Landscape-phone breakpoint (`@media (max-height: 560px)`)** is the single hook for short viewports; see the detailed note in `CLAUDE.md` before touching `.title`, `.charselect`, or `.hand`/`.combat-controls` layout — CSS cascade order (not media-query nesting) decides which override wins, and `.combat-controls` uses fixed screen-relative positions that `.hand` must track with matching `%`-based `calc()` math, not `100vw`.
- **Browser testing**: install `playwright-core --no-save`, launch Chromium at `/opt/pw-browsers/chromium-1194/chrome-linux/chrome` with `--no-sandbox`. Never wait on `networkidle` (looping audio/rAF never idles) — use `domcontentloaded` + `waitForTimeout`. `window.__ase` is the live Game instance for asserting run state **and driving scenes**: `__ase.showTitle()`, `__ase.showCharSelect()`, `__ase.startRun('amara')`, `__ase.showMap()`, `__ase.startMonster()`. Set `localStorage` `spire_of_ase_meta_v1` `tutorialDone:true` before combat so the first-play tutorial doesn't block screenshots.
- **QA screenshot / responsiveness audit**: `npm run qa` (`node tools/qa-screenshots.js`) drives title → char select → map → combat at a landscape phone (812×375) and a desktop (1366×850) viewport, writes JPEG screenshots + `report.md`/`report.json` to `docs/qa/`, and flags horizontal overflow, off-screen/clipped controls, and hand cards overlapping the energy orb / End Turn. Clean run = `0 issue(s)`; committed `docs/qa/*.jpg` are the baseline. It found & we fixed a real desktop-combat bug: the fanned hand was cropped at the bottom and slid under the corner controls. Fix lived in the base `.hand`/`.card.in-hand` rules (bottom padding 30px + dip cap `min(--shift,18px)`) plus the desktop breakpoint (safe-zone margins **and** a smaller `--card-h` cap — margins alone can't help because `.hand` is `justify-content:center`, so the whole rotated hand must actually fit the band between the controls).
- **Combat transitions / pacing** (added here): node entry runs through `Game.veilTransition(swapFn)` (a `.scene-veil` fixed fade veil) so combat/events don't snap in. Combat **opens deferred** — `CombatView.mount` shows a "Battle Start" banner then calls `combat.start()` ~650ms later, so the opening draw deals card-by-card *after* the popup; the tutorial kickoff in `beginCombat` (~1700ms) is timed to that, so keep them in sync. Two cosmetic engine events drive the pacing text: `fx('announce',{text,kind})` → `CombatView.announce()` centered banner (`battle`/`player`/`enemy` = Battle Start / Your Turn / Enemy Turn), and `fx('enemyMove',{source,name})` → move name floated over the enemy via `floatText(...,'name')`. `combat.js`'s `enemyPhase` **`await`s** after the Enemy-Turn banner (850ms), after each move-name float (750ms), and after each action resolves (550ms) so the strike never lands the same instant it's announced — preserve those waits. `endTurn` adds one more `await wait(600)` after `enemyPhase()` returns and before `startPlayerTurn(false)`, so "Your Turn" doesn't fire in the same instant the last enemy's hit lands (skipped when `this.over`). Card play is **two-tap**: first tap sets `previewCard` (card straightens + enlarges via `.previewing`; `--angle/--shift` zeroed in `renderHand`), second tap commits; drag-to-play bypasses the preview. All of it honors `prefers-reduced-motion` (banner/veil have fade fallbacks). **Gotcha**: `npm run qa` wipes `docs/qa/`, deleting the committed `docs/qa/README.md` — `git checkout -- docs/qa/README.md` before committing.
- **`Combat.notify()` coalescing (real bug found & fixed here)**: several call sites fired two synchronous `notify()`s in the same tick (e.g. `startPlayerTurn`'s `draw()` fires one internally, then the method fires another right after to refresh the log). `CombatView.renderHand()` fully rebuilds the hand's DOM on every call, so the second same-tick render destroyed the first render's newly-hidden new-card nodes before their `requestAnimationFrame`-scheduled draw-pile fly-in animation could run — cards ended up in the right spot but silently teleported instead of animated, on every draw (including the opening hand) since the fly-in was added. Fixed by making `notify()` coalesce same-tick calls via a microtask (`Promise.resolve().then(...)`, guarded by `_notifyScheduled`) into one deferred `onUpdate()`. Because of this, `tools/qa-screenshots.js`'s `combat` scene `settle` had to go from 1100ms to 1800ms — the fly-in now actually takes ~830ms after the 650ms deferred `combat.start()` to fully land. If a per-card/per-element cosmetic animation you add silently doesn't play despite the underlying state change happening, suspect a same-tick double-render stomping a pending `requestAnimationFrame` before you reach for anything more exotic.
- **Animated background** (`src/fx/background.js`) exports a `background()` singleton controller: `setAct(act)`, `setCombat(on)`, `pulse(kind, power)`. It's act-themed (palette per act) and reacts to combat FX; keep new pulses reduced-motion-safe.
- **Ascension** lives in `core/state.js` (`ASCENSION_LEVELS`, `MAX_ASCENSION`, `run.enemyHpMult`, `run.enemyDamageMult`, `run.restHealFraction`). Levels are cumulative; unlocked one at a time by winning at the current max. Meta keys: `maxAscension` (unlocked) and `ascension` (last selected). The ascension picker in `game.js` renders straight from `ASCENSION_LEVELS`, so adding a level is just appending an entry there plus wiring its modifier at the right hook (HP → `enemyHpMult`, per-hit damage → `enemyDamageMult` applied in `combat.enemyAttack`, one-off run/combat setup → `RunState` constructor or `combat.start()`).
- **Story framing = "the Spire eats its winners"**: ascension is extraction disguised as apotheosis; enemies are former champions (their identity is baked into `tools/sprites.manifest.json` art prompts, not icons — re-run `node tools/gen-sprites.js --ids <id>` to see it). **Two endings** in `game.js` `victory()`: the first-ever clear (`meta.ascendedOnce`) is always the complicit `endVictory('ascend')`; a later clear reached **without** the `ascendant_crown` relic triggers `showEndChoice()` → *Unwrite* true ending (`meta.spireUnwritten`). The Crown is a real boss reward now (`pickBossRelicId` + `bossrelic` row) — don't re-exclude boss relics. New meta flags in `save.js` `defaultMeta`.
- **Card-choice ("pick a card") popup** (`Game.cardChoiceOverlay` in `game.js`, `.card-picker` in `styles.css`) uses fixed-aspect-ratio cards (`height:/width: var(--card-h)/var(--card-w)`, not the shop/deck overlays' `height:auto;min-height:` grow-to-fit-text trick) so options read as normal cards instead of stretched slabs with dead space. `--card-h` is `clamp(232px, 25vw, 320px)` on `.card-picker` — floors high enough on narrow phones that even the longest card text (`the_long_song`, ~106 chars) fits; `.card-desc` is top-aligned (`align-items:flex-start`) there so if a description *does* overflow it trims cleanly off the bottom instead of a centered clip eating both the start and end of the sentence. Selection is two-tap: first tap adds `.selected` (translateY + scale + glow, reusing the pre-existing but previously-unused `.card.selected` rule) and — since touch never fires `mouseenter` — also calls `this.tooltip(c, node, true, 'card')` directly from the click handler so the full, unclipped text is always reachable; tapping the same card again confirms and closes the overlay. Tapping a different card switches the selection instead of confirming. `.overlay-box`'s existing `overflow-y:auto`/`max-height:88vh` already makes the whole popup scrollable, which is what saves the layout on short landscape phones (812×375) where the fixed card height doesn't leave room for the hint + Skip button — confirmed via `box.scrollTop` in a Playwright check, not a new mechanism.
- **Per-turn enemy damage cap ("Invincibility")**: a data-driven field `dmgCapPerTurn` on an enemy blueprint (see `heart_of_static`) caps total damage that enemy can take across a single *player* turn. `combat.makeEnemy` seeds `dmgTakenThisTurn`, `applyDamage` clamps + banks pre-block damage (emits a `warded` fx), and `startPlayerTurn` resets the counter. `combat.start()` mirrors the value into `powers.invincibility` purely so the pip shows the cap — it's display-only, never ticked. `strengthDown` ("Resolve Down") is a real player debuff now: enemies apply it (`void_chanter` "Unname", `sand_wraith` "Wither"), and `tickTurnDebuffs` saps `strength` by the stacked amount at end of turn, then clears it. When adding an enemy `pick`, prefer `rng.weighted([{value,weight}])` + the player-state helpers (`playerLowHp`, `playerBlocked`, `playerLacks`, `selfLowHp` in `enemies.js`) over fixed `turn % n` cycles.
- **Never use emoji in-game.** All iconography is custom SVG line-art in `src/ui/icons.js` (or a real sprite via `src/ui/sprites.js`) — no emoji characters in data files (`enemies.js`, `characters.js`, `keywords.js`) or UI code, even as unused/dead fields. The one real bug found and fixed here: `combatView.js`'s `power` fx handler was calling `floatText(..., def.icon, ...)`, rendering the raw emoji from `POWERS[key].icon` as a floating combat number. Fixed by adding `floatHTML()` to `fx.js` and rendering `<i class="pip-ic">${powerIcon(key)}</i>` (the existing SVG icon helper, same pattern already used for on-board status pips) instead. Typographic dingbats (`❖ ❘ ✦` in `game.js`, used as plain decorative dividers) and emoji in `tools/gen-*.js` console-log output are out of scope — they're not pictographic and never reach the player.
- **CSS comment trap that silently ate a whole `@media` block for a long time**: never let an asterisk and a slash land next to each other inside a `/* ... */` comment (e.g. writing `--card-*/--med` to mean "the `--card-*` and `--med` custom properties"). The `*/` closes the comment early; everything after it up to the next `{` — including the real `@media (...) {` that follows — gets swallowed as the prelude of one bogus, invalid selector, and the browser silently discards the whole rule (it won't error in the console). This is invisible in a text diff and doesn't break `node --check` (that's JS, not CSS) — the only way to catch it is to actually inspect parsed rules (e.g. `document.styleSheets` / `CSSMediaRule` in a live page) or eyeball the rendered layout at that breakpoint. `@media (max-width: 760px)` in `styles.css` (the narrow-portrait-phone rules: battlefield single-column, char-grid single-column, `.map-legend`/`.map-title` sizing, etc.) was dropped this way for a long time before being fixed — if a breakpoint's styles ever seem to have zero effect despite matching the media condition, suspect this before anything else.
- **Map (`showMap()` in `game.js` + the "map" section of `styles.css`)**: nodes are plain line-art glyphs (`NODE`/`NODE_ICON` in `src/ui/icons.js`) with no circular badge behind them — sizing is entirely on the `.map-node` wrapper div (`.map-node .svg-ic` just fills most of it), so "bigger node" is "bigger wrapper," and `.map-node.current`/`.node-boss` just use larger width/height rather than a `transform: scale()` (scale would fight the `:hover`/pulse transforms, which only ever translate/scale from the *reachable* state). Each node's on-board position gets a small deterministic pixel jitter (`jitter(row, col)` in `showMap()`, hashed from row/col — not `Math.random()`, so it's stable across re-renders and the SVG edges — drawn from the same `posOf()` helper — always terminate exactly on the jittered icon center) so the layout reads hand-drawn instead of gridded. The map scene is a fixed-height flex column (`.scene.map { height:100dvh; overflow:hidden }` → `.map-scene` flex:1 → `.map-scroller` flex:1 with its own `overflow-y:auto`) so only the inner scroller scrolls and the board always reaches the bottom of the viewport; using `min-height` instead of `height` here breaks this — the flex item is then free to grow past the container instead of scrolling internally, and the whole *page* scrolls instead (topbar and header included). **Art-direction pass (2026-07):** there is no headline/`​.map-header` anymore (act name lives only in the top bar); the act backdrop shows at `opacity:0.2` (`body.scene-map #bg-image`) with a more translucent `.map-scroller` gradient so the painting reads behind the board. The legend is an **always-on framed panel** (`.map-legend` `<aside>` built in `showMap()`) pinned to the right of the board inside a `.map-body` flex row (scroller flex:1 + fixed-width legend); it has a `.map-legend-head` title + `.map-legend-list` of simple icon+label rows (no descriptions) and an ornate double-border/corner-tick frame. On `max-width:760px` `.map-body` stacks to a column and the legend reflows to a compact horizontal chip strip below the board; on `max-height:560px` it stays a trimmed right column. Per-edge connector ornaments (a faint `.edge-dot` diamond at the midpoint + a `.edge-arrow` chevron rotated toward the target at ~72%) and node accents (concentric ring `::before` on `.current`/`.node-boss`, twinkling sparkle `::after` on reachable + boss, disabled under reduced motion) are all deliberately subtle. The top-bar `.tb-fs`/`.tb-mute` buttons are circular with a faint inset ring — global chrome, shown on every scene, so QA's combat shot guards against regressions.
