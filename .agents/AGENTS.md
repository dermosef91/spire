# Rules for √Ä·π¢·∫∏ Development

- Always use the `gpt-image-2` model for all image, sprite, and background art generations.
- Never use Gemini or any other image models for generating assets.
- **Syntax Check**: Before committing or deploying, always verify modified JavaScript files using `node --check <path_to_file>` to catch syntax errors like unclosed blocks or brackets.
- **Events ("?" nodes)**: Story events live in `src/data/events.js` and must be original to the √Ä·π¢·∫∏ world (Sunken Market / Brass Archive / Static Crown) ‚Äî never reskins of Slay the Spire events. Each event has an `art` id; the view (`eventArt()` in `game.js`) shows `assets/event-art/<art>.png`, falling back to a committed `.svg` placeholder, then the generic "?" glyph. Generate art with `node tools/gen-event-art.js` (reads `tools/events.manifest.json`); `--dry-run` writes SVG placeholders with ZERO deps.
- **First-play tutorial**: `src/ui/tutorial.js` (`CombatTutorial`) is a single top-pinned coaching banner shown once, on the first-ever combat, gated by `meta.tutorialDone` (set true when finished OR skipped). Started from `beginCombat` in `game.js` ~800ms after mount. It reads live combat state by chaining `combat.onUpdate` (never mutates game logic) and never blocks the board. Steps are text + either a `button` (advance on click) or `await:'play'|'endturn'` (auto-advance when snapshotted energy/hand/turn changes). No spotlight/positioning/catchers ‚Äî kept deliberately minimal. Re-test by clearing `localStorage`. Styles under the "first-play tutorial" block in `styles.css`.
- **Auto-Update Learnings**: On every action/task, if you discover a project-specific gotcha, solve a debugging issue, or establish a new convention/pattern, you must immediately update `CLAUDE.md` and `.agents/AGENTS.md` to persist this learning.
- **Landscape-phone breakpoint (`@media (max-height: 560px)`)** is the single hook for short viewports; see the detailed note in `CLAUDE.md` before touching `.title`, `.charselect`, or `.hand`/`.combat-controls` layout ‚Äî CSS cascade order (not media-query nesting) decides which override wins, and `.combat-controls` uses fixed screen-relative positions that `.hand` must track with matching `%`-based `calc()` math, not `100vw`.
- **Browser testing**: install `playwright-core --no-save`, launch Chromium at `/opt/pw-browsers/chromium-1194/chrome-linux/chrome` with `--no-sandbox`. Never wait on `networkidle` (looping audio/rAF never idles) ‚Äî use `domcontentloaded` + `waitForTimeout`. `window.__ase` is the live Game instance for asserting run state **and driving scenes**: `__ase.showTitle()`, `__ase.showCharSelect()`, `__ase.startRun('amara')`, `__ase.showMap()`, `__ase.startMonster()`. Set `localStorage` `spire_of_ase_meta_v1` `tutorialDone:true` before combat so the first-play tutorial doesn't block screenshots.
- **QA screenshot / responsiveness audit**: `npm run qa` (`node tools/qa-screenshots.js`) drives title ‚Üí char select ‚Üí map ‚Üí combat at a landscape phone (812√ó375) and a desktop (1366√ó850) viewport, writes JPEG screenshots + `report.md`/`report.json` to `docs/qa/`, and flags horizontal overflow, off-screen/clipped controls, and hand cards overlapping the energy orb / End Turn. Clean run = `0 issue(s)`; committed `docs/qa/*.jpg` are the baseline. It found & we fixed a real desktop-combat bug: the fanned hand was cropped at the bottom and slid under the corner controls. Fix lived in the base `.hand`/`.card.in-hand` rules (bottom padding 30px + dip cap `min(--shift,18px)`) plus the desktop breakpoint (safe-zone margins **and** a smaller `--card-h` cap ‚Äî margins alone can't help because `.hand` is `justify-content:center`, so the whole rotated hand must actually fit the band between the controls).
- **Animated background** (`src/fx/background.js`) exports a `background()` singleton controller: `setAct(act)`, `setCombat(on)`, `pulse(kind, power)`. It's act-themed (palette per act) and reacts to combat FX; keep new pulses reduced-motion-safe.
- **Ascension** lives in `core/state.js` (`ASCENSION_LEVELS`, `MAX_ASCENSION`, `run.enemyHpMult`, `run.restHealFraction`). Levels are cumulative; unlocked one at a time by winning at the current max. Meta keys: `maxAscension` (unlocked) and `ascension` (last selected).
- **Card-choice ("pick a card") popup** (`Game.cardChoiceOverlay` in `game.js`, `.card-picker` in `styles.css`) uses fixed-aspect-ratio cards (`height:/width: var(--card-h)/var(--card-w)`, not the shop/deck overlays' `height:auto;min-height:` grow-to-fit-text trick) so options read as normal cards instead of stretched slabs with dead space. `--card-h` is `clamp(232px, 25vw, 320px)` on `.card-picker` ‚Äî floors high enough on narrow phones that even the longest card text (`the_long_song`, ~106 chars) fits; `.card-desc` is top-aligned (`align-items:flex-start`) there so if a description *does* overflow it trims cleanly off the bottom instead of a centered clip eating both the start and end of the sentence. Selection is two-tap: first tap adds `.selected` (translateY + scale + glow, reusing the pre-existing but previously-unused `.card.selected` rule) and ‚Äî since touch never fires `mouseenter` ‚Äî also calls `this.tooltip(c, node, true, 'card')` directly from the click handler so the full, unclipped text is always reachable; tapping the same card again confirms and closes the overlay. Tapping a different card switches the selection instead of confirming. `.overlay-box`'s existing `overflow-y:auto`/`max-height:88vh` already makes the whole popup scrollable, which is what saves the layout on short landscape phones (812√ó375) where the fixed card height doesn't leave room for the hint + Skip button ‚Äî confirmed via `box.scrollTop` in a Playwright check, not a new mechanism.
- **Never use emoji in-game.** All iconography is custom SVG line-art in `src/ui/icons.js` (or a real sprite via `src/ui/sprites.js`) ‚Äî no emoji characters in data files (`enemies.js`, `characters.js`, `keywords.js`) or UI code, even as unused/dead fields. The one real bug found and fixed here: `combatView.js`'s `power` fx handler was calling `floatText(..., def.icon, ...)`, rendering the raw emoji from `POWERS[key].icon` as a floating combat number. Fixed by adding `floatHTML()` to `fx.js` and rendering `<i class="pip-ic">${powerIcon(key)}</i>` (the existing SVG icon helper, same pattern already used for on-board status pips) instead. Typographic dingbats (`‚ùñ ‚ùò ‚ú¶` in `game.js`, used as plain decorative dividers) and emoji in `tools/gen-*.js` console-log output are out of scope ‚Äî they're not pictographic and never reach the player.
